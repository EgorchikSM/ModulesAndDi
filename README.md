# Модульное приложения на C# с DI

## Описание проекта

Этот проект демонстрирует построение расширяемого приложения на C#, где функциональность добавляется через модули без изменения ядра. Используется контейнер внедрения зависимостей (DI) для регистрации сервисов модулей. Приложение поддерживает загрузку модулей из конфигурации и внешней папки `modules`.

### Структура проекта

```
ModulesAndDi
  ├── Core           (ядро: интерфейсы IAppModule, IAppAction, ModuleCatalog)
  ├── Services       (сервисы ядра: IClock, IStorage)
  ├── Modules        (реальные модули: Core, Logging, Validation, Export, Report)
  ├── Program.cs     (логика запуска приложения и инициализации модулей)
  ├── appsettings.json
  └── Pr2.ModulesAndDi.csproj

ModulesAndDi.Tests
  └── ModuleCatalogTests.cs (тесты порядка запуска, отсутствующих и циклических модулей, DI)
```

## Инструкция запуска

1. Откройте решение `ModulesAndDi.sln` в Visual Studio 2022 (или выше).
2. Убедитесь, что проект `ModulesAndDi` установлен как стартап-проект.
3. Соберите решение.
4. Запустите проект (`F5` или `Ctrl+F5`).
   Приложение:

   * Считывает список модулей из `appsettings.json`.
   * Загружает модули из текущей сборки и из папки `modules`.
   * Строит порядок запуска с учётом зависимостей.
   * Регистрирует сервисы модулей в DI-контейнере.
   * Инициализирует модули и выполняет их действия (`IAppAction`).

Пример `appsettings.json`:

```json
{
  "Modules": [
    "Core",
    "Logging",
    "Validation",
    "Export",
    "Report"
  ]
}
```

## Демонстрационные модули

| Модуль           | Зависимости      | Действия                                     |
| ---------------- | ---------------- | -------------------------------------------- |
| CoreModule       | —                | Регистрирует `IClock` и `IStorage`           |
| LoggingModule    | Core             | Логирует сообщение в консоль                 |
| ValidationModule | Core             | Проверяет данные и сохраняет их в `IStorage` |
| ExportModule     | Core, Validation | Экспортирует данные в файл `export.txt`      |
| ReportModule     | Core, Export     | Выводит отчёт о текущих данных               |

## Набор проверок (тесты xUnit)

1. **Порядок запуска модулей**

   * Тест `Order_IsCorrect_MultipleDependencies` проверяет топологическую сортировку зависимостей.

2. **Отсутствующий модуль**

   * Тест `MissingModule_ThrowsModuleLoadException` проверяет, что приложение сообщает о недостающем модуле.

3. **Циклическая зависимость**

   * Тест `CyclicDependency_ThrowsModuleLoadException` проверяет, что приложение выявляет цикл зависимостей.

4. **DI внедрение зависимостей**

   * Тест `Dependencies_AreInjected_ByContainer` проверяет, что сервисы регистрируются и внедряются через DI, а не создаются вручную в модуле.

## Демонстрационные сценарии

### Сценарий 1: Полный набор модулей

`appsettings.json`:

```json
["Core", "Logging", "Validation", "Export", "Report"]
```

* Результат: все модули загружены, действия выполняются по порядку `Core → Logging → Validation → Export → Report`.

### Сценарий 2: Отсутствие модуля

`appsettings.json`:

```json
["Core", "Validation", "Export", "Report"]
```

* Результат: исключение `ModuleLoadException` с сообщением о том, что отсутствует `Logging`.

### Сценарий 3: Циклическая зависимость

* Временная модификация зависимостей:

  * Export зависит от Report, а Report зависит от Export.
* Результат: исключение `ModuleLoadException` с сообщением о цикле.

### Сценарий 4: Только базовое ядро

`appsettings.json`:

```json
["Core"]
```

* Результат: загружен только CoreModule, DI сервисы доступны, дополнительных действий нет.

## Архитектурные пояснения

1. **Ядро**

   * Папка `Core`: содержит интерфейсы `IAppModule`, `IAppAction`, `ModuleCatalog`.
   * Папка `Services`: сервисы ядра (`IClock`, `IStorage`) и их реализации.
   * Ядро **не знает деталей модулей**, только их контракт.

2. **Модули**

   * Папка `Modules`: реализация конкретных расширений.
   * Каждый модуль:

     * Объявляет имя и зависимости (`Requires`).
     * Регистрирует свои сервисы в DI-контейнере.
     * Выполняет инициализацию через `InitializeAsync`.
   * Новые модули можно добавлять, не меняя логику `Program.cs`. Достаточно:

     1. Создать класс, реализующий `IAppModule`.
     2. Добавить его в `appsettings.json`.
     3. (Опционально) разместить DLL в папке `modules`.

3. **DI и сервисы**

   * Все зависимости внедряются через контейнер.
   * Модули получают сервисы через `IServiceProvider`.
   * Логика ядра и модулей не смешивается: ядро лишь управляет порядком запуска и DI.

---

## Как расширять приложение

1. Создайте новый модуль `NewModule.cs` в папке `Modules`.
2. Реализуйте `IAppModule`.
3. Укажите зависимости через `Requires`.
4. Зарегистрируйте сервисы и действия через `RegisterServices`.
5. Добавьте имя модуля в `appsettings.json`.
6. Запустите приложение — новый модуль автоматически загрузится и выполнится.
